                              CHAPTER 1: INTRODUCTION

1.1 Introduction
    In recent years, the digital transformation of education systems has increased the demand for secure and reliable management of academic credentials. Educational institutions issue millions of transcripts, certificates, and degrees annually, which serve as critical tokens for employment, further study, and professional certification. However, the traditional methods of credential verification remain tethered to manual validation, centralized databases, and physical documents. These approaches are not only slow and expensive but are increasingly vulnerable to sophisticated forgery and tampering.

    [DIAGRAM PROMPT - FIG 1.1: Use Mermaid.ai to create a flow diagram showing the current manual process: Institution -> (Mails Paper) -> Student -> (Submits Physical Copy) -> Employer -> (Calls Institution) -> Verification Result. Label the delays "2-4 Weeks".]

    Academic certificate fraud is a growing global concern. The rise of "degree mills" and digital manipulation tools has made it difficult for employers to distinguish genuine credentials from counterfeits. Currently, verifying a certificate often involves weeks of communication between HR departments and university registrar offices, creating a bottleneck in the global talent market.

    To solve these systemic issues, "Credify" introduces a Blockchain-Based Verifiable Credential System. By utilizing blockchain’s immutability and transparency, the system records cryptographic fingerprints (Hashes) of documents rather than the documents themselves. This balances the need for publicly verifiable truth with the requirement for private data storage. Any modification to a transcript, no matter how minute, changes its hash value, rendering forgery mathematically impossible.

    Credify integrates:
    • Cryptographic Hashing (SHA-256): Ensuring document integrity.
    • Digital Signatures (RSA-2048): Proving institutional authenticity.
    • IPFS (InterPlanetary File System): Providing decentralized storage for high-volume data.
    • Selective Disclosure: Enhancing student privacy by revealing only required fields.

1.2 Motivation
    The motivation for Credify arises from the friction inherent in the modern academic lifecycle. Students currently have little control over their own verified data; they must request the university to "prove" their graduation repeatedly. Employers face high costs and long wait times to verify candidates, leading many to skip verification entirely—a risk that compromises organizational integrity.

    The core motivation is to empower the student. By moving to a self-sovereign identity model, students hold their own "Verifiable Credentials" in a digital wallet. This eliminates the "centralized silo" problem and enables instant, trustless verification on a global scale.

1.3 Problem Definition
    The current verification landscape is plagued by:
    1. Centralization: Single points of failure where database breaches can lead to mass credential manipulation.
    2. Latency: Manual verification cycles that delay hiring and enrollment.
    3. Forgery: High prevalence of high-quality counterfeit documents.
    4. Privacy Erosion: The "all-or-nothing" sharing model where full transcripts must be disclosed for simple verification tasks.

1.4 Objective Of The Project
    The primary objective is to build a production-grade platform called "Credify" that:
    1. Enables institutions to issue tamper-proof, digitally signed transcripts.
    2. Gives students a unified dashboard to manage and "selectively" share their credentials.
    3. Provides a public-facing portal for employers to verify credentials in seconds.
    4. Implements a decentralized storage bridge using IPFS.
    5. Produces high-fidelity, "Elite" PDF transcripts with embedded blockchain proofs.

1.5 Limitations Of The Project
    1. Simulated Consensus: The prototype uses a private blockchain model suited for institutional consortia rather than a public proof-of-work network.
    2. Key Management: Security transitions from the institution to the student's digital key management responsibility.
    3. Adoption: Real-world success relies on a network effect among universities.

                          CHAPTER 3 — LITERATURE SURVEY

3.1 Introduction
    Traditional verification relies on the "Three-Party" model (Issuer, Holder, Verifier) but lacks a common trust layer. Prior research into blockchain for education frequently struggled with "On-Chain Bloat"—trying to store large PDFs directly on the ledger.

3.2 Existing Systems
    Most existing solutions use centralized Portals (e.g., National Academic Depository). While digital, they still represent a central point of failure. Early blockchain prototypes often lacked "Selective Disclosure," forcing students to share everything or nothing.

    [DIAGRAM PROMPT - FIG 3.1: Use Mermaid.ai to show a 'Centralized' vs 'Decentralized' hub comparison. Show one big database node vs multiple nodes connected in a mesh.]

3.3 Disadvantages of Existing Systems
    1. Privacy: No control over field-level disclosure.
    2. Cost: High gas fees on public chains like Ethereum.
    3. Scalability: Slow transaction times for large institutions.

3.4 Proposed System (Credify)
    Credify solves these by using a "Hybrid Off-Chain" model.
    • Off-Chain (IPFS): Large files are stored in decentralized storage.
    • On-Chain (Blockchain): Only the Content Identifier (CID) and Hash are recorded.
    • Selective Disclosure: Students generate specific views of their data for sharing.

                      CHAPTER 4 — DESIGN AND IMPLEMENTATION

4.1 Architecture
    Credify employs a layered architecture:
    1. User Interface: Senior Grade Dashboards for Admin, Student, and Verifier.
    2. Logic Layer: Flask APIs managing the "Issue-Verify" lifecycle.
    3. Cryptography Engine: Handles RSA signing and hash linkages.
    4. Storage Bridge: Orchestrates data between the SQL DB and IPFS.

    [DIAGRAM PROMPT - FIG 4.1: High-level System Architecture diagram. Use Mermaid.ai to show WebUI -> Backend -> [SQLite, IPFS, Blockchain Ledger].]

4.2 Workflow Implementation
    1. Issuance: Admin uploads data -> System hashes JSON -> Signs with RSA -> Uploads to IPFS -> Records on Ledger.
    2. Management: Student views "Digital Records" -> Selects fields (e.g., GPA only) -> Generates Shareable Link.
    3. Verification: Employer scans QR or enters ID -> System fetches from IPFS -> Re-hashes -> Compares with Blockchain -> Verifies Institutional Signature.

                            CHAPTER 5: CONCLUSION

5.1 Conclusion
    Credify successfully bridges the gap between traditional academic administration and modern trustless technology. By implementing a private blockchain with IPFS integration, the project provides a blueprint for a tamper-proof academic record system that is efficient, privacy-preserving, and visually stunning.

5.2 Future Enhancement
